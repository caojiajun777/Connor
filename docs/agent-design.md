# Connor Daily Agent Specification v1

**Status:** Frozen  
**Date:** 2026-07-13

## Product statement

> Cursors define each account's incremental boundary only — they do not limit post count. Every new item between cursors gets one chance to compete for that day's briefing. Collection must not silently drop posts; runs must be recoverable; all candidates must be evaluable.

## Core pipeline

```text
DailyTrigger
  → acquire_run_lock
  → initialize_run          # freeze watchlist + prompt/model/top_k versions
  → load_watchlist
  → collect_accounts_loop
        per account:
          load_cursor
          → collect_until_cursor_or_boundary
          → persist posts + account_run + cursor_sync_outbox
          → PostgreSQL COMMIT
          → sync Redis from outbox
  → cursor_sync_gate
  → collection_gate
  → freeze_candidate_snapshot   # run_posts
  → summarize_all_candidates
  → summary_gate
  → evaluate_all_candidates
  → evaluation_gate
  → select_top_k                # deterministic ORDER BY
  → editorial_final_selection   # Top ≤ 20
  → persist_selection
  → finalize_run                # outbox empty or explicit partial
  → release_run_lock
```

LangGraph owns the harness from day one (thin graph in M3a; production checkpointer/scheduler in M3e). Stages are fixed — not free-form ReAct.

## Cursor eligibility (blocking rule)

```text
cursor_eligible = true
  - account original posts
  - account replies
  - account quote posts
  - other timeline entries with a post_id generated by that account

cursor_eligible = false
  - bare reposts
  - recommended / injected content
  - old pinned posts
  - quoted originals expanded from a quote card
```

Bare reposts **may** be collected, stored, summarized, and ranked.  
They **must not** set `cursor_after` or `cursor_reached`.

Preferred MCP field: stable `timeline_entry_id`. Until available, only `cursor_eligible=true` items act as cursor anchors.

**`cursor_after`:** the newest successfully persisted **`cursor_eligible=true`** timeline entry for that account in this run (not page-first, not max numeric `post_id`, not pins).

- No new eligible posts and old cursor hit exactly → `cursor_after = cursor_before`
- `page_incomplete` / `safety_limit_reached` / unaccepted `known_data_gap` → posts may persist; **do not** advance cursor

### Scan rules

1. Exact `post_id == old_cursor.post_id` on an eligible item → stop (old cursor itself not re-ingested)
2. No old cursor → chase up to 72h; all in-window posts are first-run increments
3. Old cursor missing:
   - If `last_success_at >= now - 72h` and 72h boundary covered → `cursor_not_found_but_window_covered` (advance + warning)
   - If `last_success_at < now - 72h` → **`known_data_gap`**: pause / explicit `accept_gap` / backfill — never silent advance

### Safety caps (not business truncation)

```yaml
collection:
  max_pages_per_account: 20
  max_new_posts_safety_limit: 200
```

Hitting the new-post safety limit → `safety_limit_reached`, cursor not advanced.

There is **no** per-account “max 10 into summary/candidates” rule.

## Storage

### Redis (working cursor, no TTL)

Key: `connor:x:cursor:<handle>`

```json
{
  "post_id": "...",
  "published_at": "...",
  "last_success_at": "...",
  "source_run_id": "..."
}
```

Optional short-lived `post_id` cache may use TTL; cursor keys must not.

Recovery: Redis → latest successful `account_runs.cursor_after` / pending outbox → else first-run.

### PostgreSQL (source of truth)

| Table | Role |
|-------|------|
| `runs` | Run metadata + frozen versions + gate/coverage fields |
| `account_runs` | Per-account collection + cursor snapshot |
| `cursor_sync_outbox` | Reliable PG → Redis cursor sync |
| `posts` | Durable post facts; `first_ingest_run_id` audit only |
| `run_posts` | Per-run discovery / candidate freeze |
| `post_summaries` | Versioned summaries |
| `post_evaluations` | Absolute scores; binds `summary_id` |
| `selection_runs` / `selection_items` | Final Top ≤ 20 (**≠ published**) |

Later: `report` / `report_items` / `publication_status`.

### Account write order + outbox

Inside one account transaction write `posts`, `run_posts`, `account_runs`, `cursor_sync_outbox`, then **COMMIT**, then sync Redis and mark outbox `synced`.

- Crash after COMMIT → resume drains outbox  
- Transient Redis failure → summarization may proceed; **finalize** requires outbox drained or explicit partial  
- Never update Redis before PG commit  

### Run lock

PostgreSQL **session-level** advisory lock on a dedicated connection for the whole daily run.

- Lock key: `connor_daily_pipeline`  
- Release in `finally`; process crash releases with connection  
- Future multi-watchlist: `pipeline_name + watchlist_id`

## `initialize_run` freezes versions

Frozen for the run (config/prompt edits mid-run must not apply):

```text
watchlist_hash
summary_model / summary_prompt_version / summary_prompt_hash
evaluation_model / evaluation_prompt_version / evaluation_prompt_hash
editorial_model / editorial_prompt_version / editorial_prompt_hash
top_k / top_n
```

`post_evaluations.summary_id` must point at the summary row used for that evaluation — never “latest summary for post”.

## `run_posts` idempotency

```text
UNIQUE(run_id, post_id)

posts:     upsert by post_id (global)
run_posts: upsert by (run_id, post_id) (this run's discovery)
```

```text
is_new_global   — first time this post_id enters Connor DB
is_new_for_run  — first time this run processes it
is_candidate    — enters this run's summarize + evaluate path
```

Normal path: all valid increments in the cursor interval → `is_candidate=true`.  
Migrate abandoned runs via explicit `requeue_candidates(old_run_id, new_run_id)` without changing `first_ingest_run_id`.

Candidate query:

```sql
run_posts.run_id = :current_run_id
AND run_posts.is_candidate = true
AND bound post_summaries.status = 'success'
```

## Summary & evaluation gates (symmetric)

Statuses: `pending | processing | success | failed_retryable | failed_permanent`

Default: every candidate for the next stage reaches a terminal state.

- `failed_retryable` → retry  
- retries exhausted → pause  
- explicit `accept_partial` → continue with coverage + `missing_*_post_ids` + `selection_status=partial`  

Do not silently drop failed summaries/evaluations when picking Top 20.  
Summary failure does **not** roll back account cursors.

Summaries live in versioned `post_summaries` (`model`, `prompt_version`, `prompt_hash`, …). `posts.summary_status` is only a convenience field.

## Selection

Always the same path:

```text
absolute evaluate all candidates → post_evaluations
→ programmatic Top K
→ editorial final selection → Top ≤ 20
→ selection_items
```

```text
top_k = min(candidate_count, 50)
```

Deterministic tie-break:

```text
ORDER BY
  importance_score DESC,
  information_gain_score DESC,
  specificity_score DESC,
  frontier_score DESC,
  published_at DESC,
  post_id DESC
```

```text
selection_status:   selected | not_selected
publication_status: unpublished | published | withdrawn
```

Selection success ≠ published. Report/publish failures must not mark posts published.

## Relationship to existing M1/M2

**Keep:** X MCP, session reuse, watchlist YAML, clean contextual fields, editorial code evolved into evaluate + final select.

**Daily path replaces:** whole-window file re-fetch as primary increment, per-account business retain caps, single LLM pass over raw full text, file cursors as production state.

Local CLI / fixtures remain for offline debug. Production truth is PostgreSQL + Redis.

## Acceptance checklist

```text
[ ] Cursors define incremental boundaries only
[ ] Reposts / pins / recommended / quoted originals are not cursor anchors
[ ] Per-account: persist PG (+ outbox) then Redis
[ ] All increments enter summarize + evaluate (no per-account business cap)
[ ] PostgreSQL = durable facts; Redis working cursors have no TTL
[ ] initialize_run freezes watchlist + all prompt/model/top_k versions
[ ] run_posts freezes candidates; requeue without rewriting first_ingest_run_id
[ ] post_summaries are versioned; evaluations bind summary_id
[ ] post_evaluations cover all candidates; Top K sort is deterministic
[ ] selection_items = Top 20 ≠ publication
[ ] summary_gate and evaluation_gate are symmetric; partial is explicit

```
